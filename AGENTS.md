# 项目级 AGENTS 规范

本文件定义 `D:\Qsh的个人资料\科研\LLM\My_first_project` 仓库内的项目级协作与编码规范。除非用户在当前任务中显式覆盖，否则后续工作默认遵守本规范。

## 1. 初始化与上下文恢复

1. 新会话进入本仓库时，优先读取根目录 `CONTEXT.md`，将其作为当前仓库状态、术语和主线入口的第一参考。
2. 当 `CONTEXT.md` 与旧聊天记忆、口头描述或历史设计稿冲突时，以当前代码和当前 `CONTEXT.md` 为准；若两者仍冲突，再以当前代码真实实现为准。
3. 讨论主线任务时，优先按 `CONTEXT.md` 中定义的入口脚本、结果目录和术语来理解用户意图，不要把后续未落地的 workflow 设计套回当前仓库。

## 2. 文档同步要求

1. 当你更改仓库中的代码、脚本入口、目录结构、结果读取方式、数据约定或核心术语时，必须同步更新根目录 `CONTEXT.md`。
2. `CONTEXT.md` 需要记录“当前真实状态”，不要写与当前代码不一致的规划性描述来替代事实。
3. 如果本次改动不影响仓库理解层面的上下文，可只做最小必要更新；但不能在明知上下文已变化时跳过更新。

## 3. 代码组织要求

1. 函数、模块、类必须细粒度拆分，职责单一。
2. 单个函数实现代码不得超过 100 行；接近该上限时必须继续拆成更小的辅助函数。
3. 模块划分要足够仔细，避免把无关职责混在同一文件中。
4. 主函数和脚本入口应尽量只负责编排，具体逻辑下沉到可复用的小函数。
5. 优先选择可组合的小 helper、窄接口和清晰数据流，不要为了省事堆大函数或大模块。

## 4. 注释与沟通语言

1. 与用户对话默认使用中文。
2. 代码注释默认使用中文。
3. 每个新脚本必须在文件开头写注释，说明：
   - 脚本在项目中的作用
   - 依赖或调用的其他脚本接口
   - 输入数据来源
   - 输出内容
   - 对外提供的接口或入口参数
4. 每个函数开头也必须写注释，说明：
   - 函数目的
   - 输入参数来源或含义
   - 输出结果
   - 依赖关系、调用关系或副作用

## 5. 命令行与配置约定

1. 后续新增或补充正式运行命令时，默认提供后台运行版本；Linux 侧优先使用 `nohup ... > logs/... 2>&1 &` 一类可直接落地的形式，Windows 侧如需等价写法再单独说明。
2. 命令行示例默认直接写出关键路径、数据集名和超参数，不要依赖临时 shell 环境变量去拼接主要参数，除非底层工具本身强制要求环境变量。
3. 当路径、数据集名、超参数或输出目录存在一组稳定默认值时，允许在脚本同目录增加 `config.py` 统一维护默认值。
4. `config.py` 只负责承载默认值与轻量解析辅助，不替代命令行参数；脚本入口仍需保留显式 CLI 参数，并保证命令行传参优先级高于 `config.py` 默认值。
5. 如果同一实验目录下有多个脚本共享同一批默认配置，优先复用同一个 `config.py`，避免把相同常量散落到多个入口脚本里。

## 6. 当前仓库的默认工作方式

1. 主代码目录默认是 `TLM/`，主线任务优先围绕 `requested TTL serial suite`、clean eval、controlled eval 和离线 `safety-eval` 重评分来理解。
2. `vallina` / `villina_mixed` 属于当前仓库里的专用支线术语，沿用仓库现有拼写，不自行“纠正”命名，除非用户明确要求统一重构。
3. 当前仓库没有稳定可用的统一 workflow runner；回答和改动时不要默认引入不存在的 workflow 主接口。
4. Windows 环境主要用于读代码、改代码、做 smoke、查结果；正式运行平台默认按 Linux 语境理解。

## 7. 改动原则

1. 修改前先确认当前实现和入口，不基于记忆直接下结论。
2. 不要在未核实的情况下改动主线命令、结果路径、数据集约定和评测口径。
3. 如果新增脚本、重构模块或调整接口，除改动代码外，还要补齐必要注释和 `CONTEXT.md` 记录。
4. 若用户只要求读取上下文或初始化规范，也应将本文件与 `CONTEXT.md` 维持在一致状态。

## 8. Do_as_I_do Attack 约定

1. 本轮 `Do_as_I_do` 实验的工作区统一放在仓库根目录 `do_as_I_do/` 下，而不是 `TLM/` 目录内部。
2. 数据构造脚本统一放在 `do_as_I_do/scripts/build_data/`，训练启动脚本统一放在 `do_as_I_do/scripts/train/`，预测启动脚本统一放在 `do_as_I_do/scripts/predict/`，构造后的数据统一输出到 `do_as_I_do/data/`。
3. 本轮训练 YAML 统一放在 `do_as_I_do/examples/train/`，预测 YAML 统一放在 `do_as_I_do/examples/predict/`。
4. 本轮 `Do_as_I_do` 实验不使用共享 `config.py`；每个执行脚本都应在文件开头维护自己的顶层 `CONFIG` 字典，集中保存该脚本的默认输入输出路径与文本常量。
5. 新增 `Do_as_I_do` 训练或生成脚本时，命令行参数必须直接传具体值，不使用环境变量拼接主要参数。
6. `Do_as_I_do` 数据构造脚本产出的 JSON 默认应收敛为 `instruction`、`input`、`output` 三字段格式，便于后续直接接 YAML 训练与预测。
7. `Do_as_I_do` 脚本四默认产出 12 份预测 YAML，并通过 `do_as_I_do/examples/predict/predict_yaml_manifest.json` 管理串行执行顺序；预测结果统一写入 `do_as_I_do/saves/predict/<模型别名>/<输出数据集名>/`。
